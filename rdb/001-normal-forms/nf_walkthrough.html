1つのテーブルから始める正規化ウォークスルー
非正規 → 1NF → 2NF → 3NF → BCNF → 4NF ── 同じデータを段階的に正規化していく
STEP 0 出発点 ── 全部入りの非正規テーブル

ある会社の「社員研修管理」をイメージしてください。
担当者は会社の社員で、研修には講師がいて、講師は専門科目を持ちます。社員にはスキルと趣味が複数あります。

何がまずい？
あらゆる問題がここに詰まっている：カンマ区切りの複数値、重複データ、連鎖的な依存関係…
これを1つずつ解消していきます。
1NF 値をバラせ ── 1セルに1値
問題：セルに複数値
「スキル」「趣味」列にカンマ区切りで複数値が入っている。
→ WHERE skill = 'Go' で検索できない。INDEX も効かない。
スキル = "Go, Rust" → 1セルに複数の意味が混在
↓ 行に展開する

達成
全セルがアトミック値になった。ただし行が爆発し、大量の重複が残っている。
2NF 部分依存を消せ ── PKの一部だけで決まる列を分離
問題：部分関数従属
PK = (社員ID, 講師名, スキル, 趣味) としたとき、
「社員名」「部署ID」「部署名」は 社員IDだけ で決まる。PKの一部にしか依存していない。
→ 社員名を変更するとき、何十行も更新が必要（更新異常）。
PK = (社員ID, 講師名, スキル, 趣味)
社員ID → 社員名, 部署ID, 部署名 ← PKの一部だけで決まる（部分依存）
講師名 → 科目 ← PKの一部だけで決まる（部分依存）
↓ 部分依存する列を別テーブルへ分離

3NF 推移依存を消せ ── 非キー列が別の非キー列を決めるな
問題：推移的関数従属
社員テーブルで、社員ID → 部署ID → 部署名 という連鎖がある。
「部署名」はPKから直接ではなく、非キー列「部署ID」を経由して決まっている。
→ 部署名を変更するとき、その部署の全社員の行を更新する必要がある。
社員テーブル: PK = 社員ID
社員ID → 部署ID → 部署名 ← 非キー列(部署ID)が別の非キー列(部署名)を決めている
↓ 推移依存する列を別テーブルへ

達成
すべての非キー列がPKに直接依存するようになった。部署名の変更は1行だけでOK。
BCNF 決定項＝キーにしろ ── 非キーが何かを決定するな
問題：非キーの決定項
受講テーブルを見てみましょう。PK = (社員ID, 講師名) ですが、
「講師名 → 科目」という依存がもしこのテーブル内にあったら（科目列を含む場合）、
講師名は非キー（PKの一部ではあるがスーパーキーではない）なのに他の列を決定している = BCNF違反。
ただし 2NF のステップで既に講師テーブルを分離済みなので、今回は既にBCNFを満たしています。
受講テーブル: PK = (社員ID, 講師名) → 成績 ✓ OK
講師テーブル: PK = 講師名 → 科目 ✓ OK（決定項=PK）

すべての決定項がスーパーキーになっている → BCNF達成
達成（今回は2NFの分離で自然に満たされた）
BCNFは3NFの強化版。「すべての関数従属 X → Y において、Xがスーパーキーである」が条件。
3NFでは候補キーの一部が決定項になることを許すが、BCNFは許さない。
4NF 独立な多値を混ぜるな ── 直積爆発を防げ
問題：もし分離していなかったら？
仮にスキルと趣味を1つのテーブルにまとめていたとしましょう。
スキルと趣味は互いに独立なのに、同じテーブルにあると直積（掛け算）で行が爆発します。
❌ もし「社員_スキル_趣味」テーブルがあったら

社員ID →→ 趣味（社員が持つ趣味の集合）
2つの多値依存が独立 → 直積で行が爆発する（2×2=4行）
↓ 独立な多値ごとにテーブルを分離（2NFで実施済み）

達成
独立した多値依存がそれぞれ別テーブルに分離されている。
2行 + 2行 = 4行で済み、直積の4行より意味的にも正確。
